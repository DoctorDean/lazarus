03

How Lazarus works

Clone → read the paper for intent → build/run/read-traceback/repair in a sandbox → emit a fixed integration contract: an importable module, a CLI, a pinned container, and a smoke test that encodes your sanity check.

Sandbox
Disposable container; expensive successes are snapshotted so a later failure never re-pays the build.
Commit-era pinner
Reconstructs the dependency universe as it was on the repo's last commit, from the PyPI timeline.
Repair loop
build → run → read traceback → patch → retry, bounded, isolated to the container.
Capability locator
Finds where "input → the famous output" happens and carves the minimal path to it.
Contract emitter
Module + CLI + pinned container + smoke test — verified callable on its own.

Runs on a laptop; executes anywhere. Both resurrections were driven from a MacBook and executed on a native-x86 workstation over one --docker-host ssh:// flag — because these TF binaries need native x86 that Apple-silicon emulation can't provide. The sandbox is host-pluggable: local, a lab box, a cloud VM, or a GPU rental.
